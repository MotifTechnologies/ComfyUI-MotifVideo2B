# /issue — GitHub Issue 등록

작업 중 발견한 별개 문제나 TODO를 GitHub Issue에 등록합니다.

## 사용법
- `/issue` — 직접 이슈 내용을 입력받아 등록
- `/issue 간단한 설명` — `$ARGUMENTS`를 제목으로 사용

## 프로세스

1. `$ARGUMENTS`가 비어있으면 사용자에게 "어떤 이슈를 등록할까요?" 질문합니다.
2. `gh issue list --limit 30`으로 기존 이슈를 확인합니다.
   - 같은 문제가 이미 등록되어 있으면 → 기존 이슈에 코멘트(`gh issue comment`)로 추가
   - 없으면 → 새 이슈 생성
3. 이슈 내용을 정리합니다. 아래 3가지는 반드시 포함:
   - **상황 설명**: 어떤 작업 중에, 뭘 하다가 발견했는지
   - **추정 원인 (root cause)**: 왜 이런 문제가 생겼는지 분석
   - **관련 파일/코드**: 어디서 발생했는지 (파일 경로, 함수명 등)
   - 현재 작업 중인 플랜이 있으면 `.plans/` 경로도 참고로 넣습니다.
4. 사용자에게 보여주고 확인받습니다:
```
📝 이슈 등록 (신규 / 기존 #번호에 추가)
제목: {제목}
상황: {한 줄 요약}
추정 원인: {한 줄 요약}
관련: {파일 경로}
등록할까요?
```
5. OK하면 실행합니다:
   - 신규: `gh issue create --title "제목" --body "본문"`
   - 기존: `gh issue comment 번호 --body "본문"`
6. 등록된 이슈 번호를 알려주고 원래 작업으로 돌아갑니다.

## 이슈 조회
사용자가 "이슈 목록 보여줘" 등으로 조회를 요청하면:
1. `gh issue list --limit 30`으로 열린 이슈 목록을 가져옵니다.
2. **관련성 기반 그룹핑 분석**을 수행하여 아래 포맷으로 보여줍니다:
```
📋 열린 이슈 — 그룹 분석 ({n}개)

🔧 그룹 A: {그룹명} (난이도: {낮음|중|높음} | 중요도: {낮음|중|높음} | ~{N}줄)
  #{번호} {제목}
  #{번호} {제목}
  → {함께 처리 가능 여부, 종속성 정보}

🧪 그룹 B: {그룹명} (난이도: {낮음|중|높음} | 중요도: {낮음|중|높음} | ~{N}줄)
  #{번호} {제목}

🚀 그룹 C: 독립 기능 (각각 별도 플랜)
  #{번호} {제목} (난이도: {낮음|중|높음} | ~{N}줄)

💡 추천: 그룹 {X} ({추천 이유: 작업량 적고 종속성 해소 등})
```
   - 그룹핑 기준: 도메인(같은 모듈/파일), 종속성(선후관계), 작업량(함께 처리 시 효율). 독립적이고 규모가 큰 이슈는 "독립 기능" 그룹으로 분리합니다.
   - 추천 그룹은 작업량이 적고, 다른 작업의 기반이 되거나 종속성을 해소하는 것을 우선합니다.
3. 사용자가 "이거 해결하자" 하면 `/task-plan`으로 넘어갑니다.

## 이슈 해결 규칙
- 이슈 해결은 사용자가 명시적으로 요청할 때만. Claude가 알아서 착수하지 않는다.
- 버그/문제 수정: 이슈 필수 → PR에 `Closes #번호`를 넣어서 merge 시 자동으로 닫히게.
- 기능 추가: 이슈 선택 → 이슈 없이 PR만 생성 가능. 관련 이슈가 있으면 `Closes #번호` 첨부.

## 주의
- `gh` CLI 로그인이 안 되어있으면 "gh auth login을 먼저 실행해주세요" 안내.
- 현재 플랜 작업을 중단하지 않는다. 등록만 하고 바로 복귀.
