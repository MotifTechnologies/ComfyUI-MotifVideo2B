# /task-plan — 작업 기억 시스템 세팅

큰 작업을 시작합니다. 아래 프로세스를 순서대로 실행하세요.

## 1단계: 현황 분석 (목표 불명확해도 반드시 먼저 수행)
- 관련 코드, 기존 `.plans/` 히스토리, `.manuals/` 문서를 먼저 탐색합니다. (생략 금지)
- 해당 도메인의 `.claude/skills/` 문서가 있으면 읽습니다.
- plan-writing skill을 참조하여 `.manuals/process/plan.md` 규칙을 따릅니다.
- `gh issue list --limit 30`으로 현재 열린 이슈 목록을 가져온 뒤, **관련성 기반 그룹핑 분석**을 수행하여 아래 포맷으로 보여줍니다. 실패 시(네트워크 단절, gh 미설치, 비공개 레포, 권한/승인 거부 등) 이슈 그룹핑 분석을 건너뛰고 "이슈 목록을 가져오지 못했습니다. 이슈 없이 진행합니다."를 안내한 후 사용자 지시 기반으로 계속합니다.
```
📋 열린 이슈 — 그룹 분석 ({n}개)

🔧 그룹 A: {그룹명} (난이도: {낮음|중|높음} | 중요도: {낮음|중|높음} | ~{N}줄)
  #{번호} {제목}
  #{번호} {제목}
  → {함께 처리 가능 여부, 종속성 정보}

🧪 그룹 B: {그룹명} (난이도: {낮음|중|높음} | 중요도: {낮음|중|높음} | ~{N}줄)
  #{번호} {제목}

🚀 그룹 C: 독립 기능 (각각 별도 플랜)
  #{번호} {제목} (난이도: {낮음|중|높음} | ~{N}줄)

💡 추천: 그룹 {X} ({추천 이유: 작업량 적고 종속성 해소 등})
```
- 난이도/중요도 판정 기준은 `planner.md` 참조 (single source of truth)
- 그룹핑 기준: 도메인(같은 모듈/파일), 종속성(선후관계), 작업량(함께 처리 시 효율). 독립적이고 규모가 큰 이슈는 "독립 기능" 그룹으로 분리합니다.
- 한 플랜의 예상 변경량이 200줄을 초과하면 분할을 권고합니다.
- 추천 그룹은 작업량이 적고, 다른 작업의 기반이 되거나 종속성을 해소하는 것을 우선합니다.
- 열린 이슈 중 이번 플랜으로 같이 닫을 수 있는 이슈를 식별합니다. 발견되면 01_plan.md에 `Closes #번호` 목록으로 기록하고, PR 생성 시 자동 포함합니다.
- 분석 완료 후 사용자의 지시 내용을 파악합니다. 목표가 불명확하면 분석 결과를 바탕으로 "현재 상태는 이렇고, 어떤 방향으로 진행할까요?"로 유도합니다. 분석 없이 "뭘 할까요?"만 묻지 마세요.
- **analyzer 연동**: 사용자 지시에 딥 분석 뉘앙스("분석해봐", "코드 분석", "구조 파악", "원인 추적" 등)가 포함된 경우 → "분석 에이전트 출동시킬까요?" 제안 후 analyzer 호출. 단순 탐색만 필요한 경우 → Explore 에이전트 사용 (기존 동작 유지). (analyzer.md 트리거 조건 4번과 연동)

## 1.5단계: Jira 연결 확인
- 사용자에게 "Jira 티켓을 연결할까요?" 질문
- Yes → Jira 프로젝트/티켓 번호 확인. 플랜 문서에 Jira 링크 기록란 포함.
- No → 스킵. HOOK:PLAN_CREATED에서도 재질문하지 않음.
- 이 단계에서 결정되면 HOOK:PLAN_CREATED의 Jira 질문은 스킵.

## 2단계: 계획 제출
분석 결과를 바탕으로 아래 내용을 사용자에게 **간결하게** 제출합니다:
- 목표 요약 (한 문장)
- 접근 방법 (왜 이 방식인지)
- 체크리스트 초안 (실행 순서)
- 예상 리스크

**이 단계에서 코드를 작성하지 마세요. 계획만 제출합니다.**

## 3단계: 브랜치 생성
사용자가 승인하면 작업 브랜치를 생성합니다.
version-control skill 참조 → `.manuals/dev/git.md` 브랜치 네이밍 규칙:
```
git checkout -b feat/YYYYMMDD-작업명
```

## 4단계: 문서 생성
`.plans/` 에 작업 폴더를 생성합니다:

```
.plans/YYYYMMDD-$ARGUMENTS/
├── 01_plan.md        # 목표, 접근방법, 전체 로드맵
├── 02_context.md     # 기술 선택 근거, 레퍼런스 링크, 제약 조건
├── 03_checklist.md   # [ ] 형식의 단계별 체크리스트
└── 04_log.md         # 시간순 작업 로그 (append-only, 빈 파일로 생성)
```

`$ARGUMENTS`가 비어있으면 사용자에게 작업명을 물어보세요.

## 5단계: 플랜 리뷰
문서 생성 후 반드시 reviewer 에이전트를 호출합니다 (플랜 리뷰 모드).
리뷰 결과에 따라 수정 후 사용자 OK를 받습니다.

## 6단계: 첫 항목 실행
사용자 승인 후 체크리스트 첫 번째 항목을 실행합니다. 한 번에 하나만.

## 동기화 규칙
플랜 문서 수정 시 반드시 01_plan.md → 02_context.md → 03_checklist.md 순서로 동기화.
3개 문서는 항상 싱크 상태를 유지해야 합니다.
